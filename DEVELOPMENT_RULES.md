# Walk Guide 開発ルール（ロボットOS設計指針）

## 🚀 プログラム修正時の必須確認事項

### ✅ 遵守事項：ロボット工学的アプローチ

#### 1. 脳（Brain）と顔（UI）の分離
- 全てのAI推論判断ロジックは WalkingBrain クラス（大脳）に集約すること。
- UIクラス（_WalkingGuideAppState）は、センサー入力の管理、大脳への解析依頼、およびアクション（表示発話）の実行に専念すること。

#### 2. センサーデータの整合性
- 画像解析（Sense）の結果は、必ず AnalysisResult などの構造化されたデータでやり取りすること。
- 生の応答文字列（Think）をUI側で複雑にパースしないこと。

#### 3. 3状態システム (Robot State Machine) の厳守
- **通常動作中** (AppState.normal)：知覚-判断-行動のメインサイクル。
- **コマンド待機中** (AppState.listening)：HMI（Human Machine Interface）の優先。
- **処理発話中** (AppState.processing)：単一タスクの完結保証。

#### 4. 冗長性と安全性の追求
- AIの判断（特に「安全」の判断）に疑問がある場合は、常に最も安全な（慎重な）指示を出すロジックを優先すること。
- 将来的な「小脳（ローカル解析）」と「大脳（クラウドAI）」のダブルチェックを意識したコード設計にすること。

### 🚫 禁止事項

#### 1. モノリス的な実装の禁止
- main.dart の UI クラス内に API 呼び出しコードを直接書かない（車輪の再発明と密結合を避けるため）。

#### 2. 非決定的な状態遷移の禁止
- 「今、ロボットが何をしているか」が不明確な状態（ゴースト状態）を作らないこと。
- 状態表示（マイクの色）と内部の状態変数は常に同期させること。

#### 3. 複雑な直接操作
- AIサービス（Gemini等）のAPI仕様変更にUI側が影響を受けないよう、必ず WalkingBrain で抽象化（ラップ）すること。

## 📈 状態遷移パターン

### 正常な状態遷移
`
normal  listening  processing  normal
`

### エラー発生時の対応
`
listening  エラー発生  normal (即座に復帰)
processing  エラー発生  normal (即座に復帰)
`

## 📝 実装時のチェックリスト

修正前に以下を必ず確認：

- [ ] 3状態システムを維持しているか？
- [ ] 新しい状態を作成していないか？
- [ ] エラー時に適切な状態に戻るか？
- [ ] 状態表示と実際の動作が一致するか？
- [ ] シンプルな実装になっているか？

## 💡 次のステップの提案方法

追加機能や改善案を提案する際は、視覚障害を持つ受取人（友人）が「何が変わるのか」を具体的にイメージでき、かつ開発が迷走しないよう、以下のフォーマットで構成すること。

### 1. 提案の原則（開発効率）
- **単一機能単位での提案**:
  1回の修正（1コミット/1プルリクエスト）で確実に実現・検証できる範囲に分割して提案する。大規模な構造変更を避け、動作の安定性を優先する。

### 2. ユーザー向け提案項目（アクセシビリティ視点）
ユーザーが導入の判断をしやすいよう、以下の4項目を必ず含めること。

#### ① 機能の概要（一行で）
- どのような新しい「能力」がアプリに追加されるのかを簡潔に説明する。

#### ② 利便性向上の理由
- 視覚障害者の利用シーンにおいて、具体的にどう「使い勝手が良くなる」のかを言語化する。

#### ③ 具体的な実装方法
- 内部的なロジック（どのクラスを修正し、どのAPIを叩くか）を簡潔に示す。

#### ④ 利用技術の背景
- 使用する技術の信頼性や開発元を明記し、安心感を与える。

## ⚠️ 修正履歴での失敗例

### ❌ v0.0.1+2 の失敗
- タイマーによる自動状態変更を追加し、複雑な検証処理を実装。
- 結果：「音声認識でエラーが発生しました」の連発。

### ❌ v0.0.1+4 の失敗
- 音声認識成功後に状態変更する複雑な仕組みを導入。
- 結果：「どうぞ」と言ったのにマイクが閉じたまま。

### ✅ v0.0.1+5 の修正
- シンプルな3状態システムに回帰し、基本動作を維持。
- 結果：安定した動作。
